## The Full QA Loop


Every QA cycle follows this sequence:


1. Risk Identification - Identify what could go wrong before testing begins
2. Risk Assessment - Use FMEA to prioritize which failures matter most
3. Test strategy - Decide how to test each risk, which test types to use, and how deep to go
4. Exit Criteria - Ship when residual risk is acceptable, not when bugs run out


> **Core mindset**
>
> "If this goes wrong - who gets hurt, how badly, and how likely is it?"
>
> Find ways to break the system, assess the potential failures, and choose appropriate test strategies.


## How This Framework Applies to Our Daily Work


Our typical QA cycle already follows this framework - it just hasn't been made explicit. Here is how each task maps to the framework:


| What we actually do | Framework section it maps to |
| --- | --- |
| Review new product / feature spec and derive test strategy | Step 1 - Find failures + Test plan (scope & risk sections) |
| Decide which existing test cases are relevant to run | Step 4 - Coverage strategy (match depth to RPN, not gut feel) |
| Run tests, review results, triage and discuss defects with dev | Step 2 - FMEA scoring + Step 3 - Mitigation strategy |
| Confirmation testing and regression testing | Step 5 - Exit criteria (all high-RPN items re-verified) |
| Sign off and create reports | Step 5 - Documenting residual risk and stakeholder acceptance |
| Lean / process improvement projects to cut redundant test cases and reduce cost | Step 4 + Step 1 - Low-RPN tests are the first to cut, with a documented reason. Target improving test cases with low coverage |


> **The biggest gap this framework fills**
>
> Currently, decisions about which tests to skip and which to use are made by team members' experience and personal skill level. This framework replaces that with a defensible rule: cut low-RPN tests first and prioritize tests for high-RPN risks.
>
>
>
>
>   Instead of "we skipped these tests because we ran out of time," we can say "we accepted these risks because RPN was below threshold." That is cost-conscious, LSS-compatible, and explainable to management.


## Step 1 - Identify Risk: How to Find Failures


Before writing a single test case, map the system by answering these six questions:


| Element | Question to ask | Test ideas it generates |
| --- | --- | --- |
| Inputs | What goes in? (data, actions, states) | Valid, invalid, empty, extreme, unexpected format |
| Outputs | What comes out? (results, side effects, data changes) | Correct result? Wrong result that looks right? No result? Side effects? |
| User flow | What's the path from entry to done? | Happy path, skip a step, out of order, abandon midway |
| Edge cases | What breaks the boundaries of each step? | Min/max values, simultaneous actions, rapid repeat |
| Dependencies | What must exist or work before this can run? | Kill each dependency one by one - what breaks? |
| Assumptions | What does the system expect the user to know or do? | Violate each assumption deliberately |


## Step 2 - Risk Assessment (FMEA)


Score each failure on three axes. Multiply them to get a Risk Priority Number (RPN).


| Axis | Question | Score guide (1-10) |
| --- | --- | --- |
| Severity (S) | How bad is it for the user if this fails? | 1 = minor inconvenience - 10 = data loss / safety issue |
| Occurrence (O) | How likely is this failure to happen? | 1 = almost never - 10 = happens constantly |
| Detectability (D) | Would we even notice if it failed? | 1 = obvious crash - 10 = silent, nobody notices |


> **Formula**
>
> RPN = Severity x Occurrence x Detectability
>
> Higher RPN = higher priority. Focus effort on the highest numbers first.


### What to do with each risk


| Decision | When to use it |
| --- | --- |
| **Fix it** | High severity - cannot ship with this defect |
| **Test it** | Medium risk - needs verification before shipping |
| **Monitor it** | Low detectability - hard to reproduce, watch in production |
| **Accept it** | Low RPN - cost to fix outweighs the risk. Document the decision. |


## Step 3 - Test Strategy


Once you know your risks, test strategy answers: how do you actually test each one? The sequence is: group risks by RPN -> assign a test approach -> select the right test types.


### Test approach by risk level


| Risk level | Approach | Test depth |
| --- | --- | --- |
| **High RPN** | Thorough - multiple test types, edge cases, repeat after fixes | Functional + Boundary + Negative + Integration |
| **Medium RPN** | Targeted - focus on the specific failure mode only | Functional + key failure modes |
| **Low RPN** | Lightweight - quick sanity check, move on | Functional only - does it basically work? |
| **Accepted risk** | None - document the decision and skip | - |


### Test types - what they are and when to use them


| Type | What it is | When to use |
| --- | --- | --- |
| Functional | Verify the feature does what it is supposed to do | Always - every risk level |
| Boundary | Test at the edges of valid input ranges | When inputs have limits or ranges |
| Negative | Deliberately give wrong or bad input to check failure behavior | When graceful failure matters |
| Exploratory | Unscripted - tester uses judgment to find unexpected issues | High-risk features, new or unfamiliar areas |
| Regression | Re-run existing tests after a change to catch unintended breakage | After any fix, update, or new feature |
| Integration | Test how features interact with each other and with the system | When dependencies or touchpoints exist |
| Confirmation | Re-test a specific defect after it has been fixed | After every bug fix - without exception |
| Smoke | Quick pass to confirm basic functionality works before deeper testing | At the start of a test cycle or after a new build |
| Reliability | Validate that the feature performs consistently under sustained or stressful conditions - includes performance, long-run, stress, and consistency testing | When stability over time matters, or when the feature is used repeatedly or under heavy load |

> **How to make a good test case**
>
> "A good test case is like a good story - easy to follow, easy to understand, and guides the tester smoothly through every step of the journey, from entry to exit."
>
>
>
>
>
> The 4 Cs of a Good Test Case
>
>
> 1. Clear - anyone can read it cold and know what to do
> 2. Complete - preconditions, steps, expected result. No gaps.
> 3. Concise - no fluff, no redundant steps
> 4. Consequential - if this test never existed, would anyone notice? If no, cut it.
>
> Ask the question: "If a new tester runs this tomorrow with zero context, will they know exactly what to do and what to look for?"


> **Optional - UX Testing**
>
> Apply when the feature is user-facing. Functional QA asks "does it work?" - UX testing asks "does it make sense to a real user?"
>
>
>
>
> Method: Walk through the user flow as a zero-context user. Narrate what you expect before each action. Every mismatch is a UX failure.
>
>
> | Lens | Question to ask |
> | --- | --- |
> | Clarity | Do I know what this is and what it does before I touch it? |
> | Feedback | Does the system tell me what is happening at each step? |
> | Recovery | If I make a mistake, can I get back easily? |
> | Expectation match | Did it do what I predicted it would do? |
> | Consistency | Does it behave the same way as similar features in the product? |


> **Optional - Mitigation Discussion with Dev**
>
> This is not a QA execution step - it is a communication tool for defect discussions with developers. When raising a high-risk defect, use this ladder to suggest the appropriate fix level:
>
>
> | Suggestion | What it means | Example |
> | --- | --- | --- |
> | Prevent | Remove the risk entirely at the design level | "Can we redesign this so invalid input is impossible?" |
> | Reduce | Lower the odds with guards or validation | "Can we add input validation before this runs?" |
> | Detect | Add monitoring or alerts so failures are caught fast | "Can we log this and alert when it happens?" |
> | Recover | Add a fallback or rollback as a last resort | "At minimum, can we show an error and let the user retry?" |
>
>
> QA identifies the risk and severity. Dev owns the mitigation decision. Use this as a starting point for conversation, not a prescription.


## Step 4 - Exit Criteria: when to Stop Testing


> **Core rule**
>
> Stop when the remaining risk is acceptable - not when you have found no more bugs. There will always be more bugs. The goal is a defensible, documented decision to ship.


### Exit criteria checklist


Before signing off on a feature, confirm all three:

- All high-risk areas (high RPN) have test coverage
- Residual risk is understood and formally accepted by a stakeholder
- Every known open bug has a decision: fix before ship / fix after ship / accept

> **Important habit**
>
> Define your exit criteria before you start testing - not when you feel like stopping. Write it down at the start of every test cycle.


## Test Case Template


Use this structure for every test case. One test case = one thing being verified. Do not bundle multiple behaviors into a single test case.


| Field | What goes in it | Example |
| --- | --- | --- |
| Test ID | Unique identifier | TC-001 |
| Title | One line - what you are testing | Login fails with incorrect password |
| Preconditions | What must be true before running this | User account exists, app is open |
| Test Workload | Numbered, in Minutes and 1 Person/Day | 480 mins = 1PD |
| Test steps | Numbered, one action per step | 1. Open login screen 2. Enter wrong password 3. Tap Login |
| Expected result | Exactly what should happen (be specific) | Error message appears within 2 seconds. No login occurs. |
| Actual result | What actually happened (fill during execution) | - |
| Status | Outcome of the test run | Pass / Fail / Blocked / Skip |
| Priority | Tied to your RPN score | High / Medium / Low |


> **Writing good expected results**
>
> Bad: "Works correctly" - too vague, cannot be verified.
>
>
> Good: "Button turns green and a success message appears within 2 seconds." - specific and verifiable.


## Test Plan - What It Is and What Goes In It


A test plan is a document written before testing begins that answers one question: "How will we verify this product is ready to ship?"


> **The key difference from a test case list**
>
> A test case list starts from existing tests and picks what seems relevant.
>
>
>   A real test plan starts from risk - then decides what tests are needed to cover it.
>
>
>   This is why things slip through checklists: if no existing test covers a new risk, the checklist misses it by design.


### The 6 sections of a test plan


| Section | Question it answers | Why it matters |
| --- | --- | --- |
| 1. Scope | What are we testing? What are we deliberately NOT testing? | Makes boundaries explicit. Prevents scope creep. |
| 2. Objectives | What does "good enough to ship" look like? | Aligns the team on definition of done before testing starts. |
| 3. Risk assessment | What are the highest-risk areas and why? | Drives where effort is focused. New features = new risks not covered by old tests. |
| 4. Test strategy | How will we test each area? (manual, exploratory, regression) | Ensures right type of testing is applied to right risk level. |
| 5. Resources & schedule | Who tests what, and by when? | Makes accountability clear and flags if timeline is unrealistic. |
| 6. Exit criteria | What conditions mean we are done? | Prevents testing from running forever or stopping too early. |


### Minimum viable Test Plan template


> **Simple test plan template**
>
> Product / Feature: [name]
>
>
> Version: [version]  -  Date: [date]  -  Author: [name]
>
>
>
>
> In scope: [list what will be tested]
>
>
> Out of scope: [list what will NOT be tested, and why]
>
>
>
>
> New risks this release: [list new features or changes + FMEA score for each]
>
>
> Test strategy: [how each risk area will be tested]
>
>
> Who / When: [owner and deadline per area]
>
>
>
>
> Exit criteria: All high-RPN items tested. All open bugs triaged. Stakeholder sign-off received.
